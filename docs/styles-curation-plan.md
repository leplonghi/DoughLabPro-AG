# Styles Library Curation Plan

## Goal
Keep `Styles` useful as a knowledge system instead of a growing pile of overlapping cards.

## Library rules
- `Flagship`: indispensable reference styles that define the main reading path of the library.
- `Core`: important supporting styles that deepen a family without diluting the main spine.
- `Specialist`: valid but non-essential default entries, shown only when explicitly revealed or searched.
- `Community`: user-created and contributed styles that follow the same template but never replace the curated backbone.

## What the default browse should optimize for
- Clear first read for a new user.
- Strong technical coverage across pizza, bread, pastry, and flatbread.
- Minimal redundancy between very similar branches.
- Easy comparison against canonical styles.

## Redundancy policy
- When two styles share the same branch, keep one as the canonical comparison anchor.
- Secondary siblings stay visible as `core` or `specialist`, not as duplicate flagships.
- Example groups:
  - `nyc_bagel` is canonical; `montreal_bagel` is the sibling comparison.
  - `french-croissant` is canonical; `pain_au_chocolat` is a shaped laminated sibling.
  - `pizza-napoletana` is canonical for benchmark pizza reading; nearby branches compare against it rather than replacing it.

## Governance checklist for every new style
1. Decide its tier before publishing it.
2. Define whether it strengthens a missing branch or duplicates an existing one.
3. Assign a canonical comparison group if it belongs to a close family.
4. Provide at least:
   - editorial summary
   - technical profile
   - process map
   - comparison target
   - integration links
5. If it cannot meet the flagship standard yet, keep it in `core` or `specialist`.
